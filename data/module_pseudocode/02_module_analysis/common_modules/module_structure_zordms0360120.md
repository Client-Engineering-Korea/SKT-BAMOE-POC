### 3.4 요금제_가입체크 (신규 가입)

**설명**: 신규 가입 시 요금제 사용 가능 여부를 체크

**입력 파라미터**:
- 요금제 객체 (상품ID, 가입해지구분 등)

**주요 처리 로직**:

#### 1. 요금제 기본 정보 조회
- 요금제마스터 테이블 조회 (요금제ID, 처리일시)
- 조건: 요금제정보 = NOT_FOUND
- 결과: 처리 불가
- 불가사유: "요금제 정보 없음"

#### 2. 판매 가능 여부 체크
- 조건: 요금제정보.판매가능여부 = "N"
- 결과: 처리 불가
- 불가사유: "판매 중지된 요금제"

#### 3. 판매 기간 체크
- 조건: 처리일자 < 판매시작일 OR 처리일자 > 판매종료일
- 결과: 처리 불가
- 불가사유: "판매 기간 아님"

#### 4. 가입 대상 체크 (나이)
- **최소나이 체크**
  - 조건: 최소나이 > 0 AND 고객.나이 < 최소나이
  - 결과: 처리 불가
  - 불가사유: "최소 [N]세 이상 가입 가능"
- **최대나이 체크**
  - 조건: 최대나이 > 0 AND 고객.나이 > 최대나이
  - 결과: 처리 불가
  - 불가사유: "최대 [N]세 이하 가입 가능"

#### 5. 채널 제한 체크
- 조건: 요금제정보.채널제한 = "Y"
- 요금제판매채널 테이블 조회 (요금제ID)
- 조건: 입력.채널 NOT IN 허용채널목록
- 결과: 처리 불가
- 불가사유: "해당 채널에서 판매 불가"

#### 6. 단말 세대 체크 (3G/4G/5G)
- 조건: 요금제정보.단말세대제한 = "Y"
- 단말정보 테이블 조회 (단말모델코드)
- 조건: 단말정보.세대 < 요금제정보.요구세대
- 결과: 처리 불가
- 불가사유: "[요구세대] 이상 단말 필요"

#### 7. USIM 타입 체크
- 조건: 요금제정보.USIM타입제한 = "Y"
- USIM정보 테이블 조회 (USIM일련번호)
- 조건: USIM정보.타입 ≠ 요금제정보.요구USIM타입
- 결과: 처리 불가
- 불가사유: "[요구USIM타입] USIM 필요"

#### 8. 지역 제한 체크
- 조건: 요금제정보.지역제한 = "Y" (예: 수도권 전용)
- 요금제판매지역 테이블 조회 (요금제ID)
- 조건: 고객.지역코드 NOT IN 허용지역목록
- 결과: 처리 불가
- 불가사유: "해당 지역에서 가입 불가"

**반환값**:
- 출력.사용가능여부: "Y" / "N"
- 출력.불가사유: 처리 불가 시 구체적 사유

**에러 처리**:
- 각 체크 단계에서 조건 미충족 시 즉시 실패 반환
- 불가사유를 명확히 설정하여 반환

### 3.5 요금제_변경체크 (요금제 변경)

**설명**: 요금제 변경 시 사용 가능 여부를 체크

**입력 파라미터**:
- 요금제 객체 (신규 요금제 정보)

**주요 처리 로직**:

#### 1. 기존 요금제 조회
- 고객현재요금제 테이블 조회 (서비스관리번호)
- 조건: 기존요금제 = NOT_FOUND
- 결과: 처리 불가
- 불가사유: "현재 가입된 요금제 없음"

#### 2. 동일 요금제 체크
- 조건: 기존요금제.요금제ID = 신규요금제ID
- 결과: 처리 불가
- 불가사유: "이미 가입한 요금제"

#### 3. 신규 요금제 가입 조건 체크
- `요금제_가입체크(요금제)` 재귀 호출
- 신규 요금제의 모든 가입 조건 확인 (Rule 1 적용)
- 조건: 출력.사용가능여부 = "N"
- 결과: 처리 불가 (가입체크의 불가사유 그대로 반환)

#### 4. 최소 사용 기간 체크
- 조건: 기존요금제.최소사용기간 > 0
- 사용기간 계산 (기존요금제.가입일 ~ 입력.처리일자)
- 조건: 사용기간 < 기존요금제.최소사용기간
- 결과: 처리 불가
- 불가사유: "최소 [N]일 사용 필요"

#### 5. 요금제 변경 횟수 제한 체크
- 요금제변경이력 테이블 조회 (서비스관리번호, 최근1개월)
- 조건: 변경이력.변경횟수 ≥ 3
- 결과: 처리 불가
- 불가사유: "월 3회 변경 제한 초과"

#### 6. 세대간 변경 제한 체크
- 신규요금제정보 테이블 조회 (신규요금제ID)
- 조건: 기존요금제.세대 = "3G" AND 신규요금제정보.세대 = "5G"
- 조건: 직접변경불가규칙 = "Y"
- 결과: 처리 불가
- 불가사유: "4G 거쳐서 변경 필요"

#### 7. 요금제 변경 제한 관계 체크
- 요금제변경제한 테이블 조회 (기존요금제ID, 신규요금제ID)
- 조건: 변경제한 EXISTS AND 변경제한.변경불가 = "Y"
- 결과: 처리 불가
- 불가사유: 변경제한.불가사유 (테이블에 정의된 사유)

#### 8. 약정 체크
- 요금제약정정보 테이블 조회 (서비스관리번호)
- 조건: 약정정보 EXISTS AND 약정정보.약정상태 = "유효"
- 경과개월수 계산 (약정시작일 ~ 처리일자)
- 남은개월수 계산 (약정개월수 - 경과개월수)
- 조건: 남은개월수 > 0
  - 위약금 계산 (`계산위약금(약정정보, 경과개월수)`)
  - 출력.위약금 설정
  - 조건: 위약금 > 0
    - 출력.경고메시지: "약정 위약금 [위약금]원 발생"

#### 9. 요금 상승/하락 체크
- **요금 상승 (30% 이상)**
  - 조건: 신규요금제정보.기본료 > 기존요금제.기본료 × 1.3
  - 출력.경고메시지: "요금이 30% 이상 상승합니다"
- **요금 하락 (30% 이상)**
  - 조건: 신규요금제정보.기본료 < 기존요금제.기본료 × 0.7
  - 출력.안내메시지: "요금이 30% 이상 절감됩니다"

**반환값**:
- 출력.사용가능여부: "Y" / "N"
- 출력.위약금: 발생하는 위약금 (있는 경우)
- 출력.경고메시지: 위약금 또는 요금 상승 안내
- 출력.안내메시지: 요금 절감 안내

### 3.7 계산위약금 (위약금 계산)

**설명**: 약정 위약금을 경과개월수에 따라 차등 계산

**입력 파라미터**:
- 약정정보 객체 (약정시작일, 약정개월수, 약정타입, 기본위약금 등)
- 경과개월수 (선택, NULL인 경우 자동 계산)

**주요 처리 로직**:

#### 1. 경과개월수 계산
```
IF 경과개월수 = NULL THEN
    경과개월수 ← 계산(약정정보.약정시작일, 현재일자)
END IF
```

#### 2. 남은개월수 계산
```
남은개월수 ← 약정정보.약정개월수 - 경과개월수
IF 남은개월수 ≤ 0 THEN
    RETURN 0
END IF
```

#### 3. 약정타입별 위약금 계산

**24개월 약정**:
| 경과개월수 | 위약금 비율 | 계산식 |
|-----------|------------|--------|
| 0 ~ 5개월 | 100% | 기본위약금 × 1.0 |
| 6 ~ 11개월 | 75% | 기본위약금 × 0.75 |
| 12 ~ 17개월 | 50% | 기본위약금 × 0.5 |
| 18 ~ 23개월 | 25% | 기본위약금 × 0.25 |

**12개월 약정**:
| 경과개월수 | 위약금 비율 | 계산식 |
|-----------|------------|--------|
| 0 ~ 5개월 | 100% | 기본위약금 × 1.0 |
| 6 ~ 11개월 | 50% | 기본위약금 × 0.5 |

**반환값**: 위약금 금액

### 3.8 p010_set_initial_val (초기값 설정)

**설명**: 출력 변수의 초기값을 설정

**처리 로직**:

## 7. 특이사항 및 주의사항

### 7.1 처리 특성
- **3가지 처리 유형**: 가입(01), 변경(02), 해지(03)
- **재귀 호출**: 변경 체크 시 가입 체크를 재귀 호출하여 신규 요금제 조건 확인
- **조기 종료**: 각 체크 단계에서 조건 미충족 시 즉시 실패 반환
- **조건부 조회**: 제한 플래그가 "Y"인 경우에만 상세 조회 수행

### 7.2 요금제 변경의 복잡성
- **이중 체크**: 기존 요금제 조건 + 신규 요금제 조건
- **변경 횟수 제한**: 월 3회 제한으로 과도한 변경 방지
- **세대간 제한**: 3G → 5G 직접 변경 불가 (4G 경유 필요)
- **요금 변동 고지**: 30% 이상 변동 시 명확한 안내
- **재귀 호출 주의**: 요금제_가입체크를 재귀 호출하므로 스택 오버플로우 주의

### 7.3 약정 위약금 계산
- **24개월 약정**: 6개월 단위로 25%씩 감소 (100% → 75% → 50% → 25% → 0%)
- **12개월 약정**: 6개월 단위로 50%씩 감소 (100% → 50% → 0%)
- **경과개월수 기반**: 약정 시작일부터 현재일까지 계산
- **차등 적용**: 경과 기간에 따라 공정하게 위약금 감소

### 7.4 부가서비스 연계
- 요금제 해지 시 가입 중인 부가서비스 자동 해지
- 해지 전 고객에게 명확한 안내 필요
- 부가서비스 건수를 경고 메시지에 포함

### 7.5 확장성 고려사항
- **신규 체크 항목 추가**
  - 각 체크 함수에 새로운 조건 추가
  - 조건부 조회 패턴 유지 (성능 최적화)

- **새로운 약정 타입**
  - `계산위약금` 함수에 새로운 약정타입 분기 추가
  - 위약금 계산 로직 확장

- **변경 제한 규칙 확장**
  - 요금제변경제한 테이블에 새로운 제한 관계 추가
  - 코드 변경 없이 데이터로 관리

### 7.6 성능 최적화 포인트
- **조건부 조회**: 제한 플래그가 "Y"인 경우에만 상세 조회
  - 예: 채널제한 = "Y"인 경우에만 요금제판매채널 조회
  - 예: 단말세대제한 = "Y"인 경우에만 단말정보 조회

- **캐싱 전략**
  - 요금제마스터, 단말정보, USIM정보 등 마스터 데이터 캐싱
  - 고객현재요금제, 약정정보 등 트랜잭션 데이터는 실시간 조회

- **변경 이력 조회 범위**
  - 최근 1개월로 범위 제한하여 성능 향상
  - 인덱스 활용 (서비스관리번호, 변경일자)

- **재귀 호출 최적화**
  - 요금제_가입체크 재귀 호출 시 이미 조회한 데이터 재사용 고려
  - 캐시 메커니즘 도입 검토

### 7.7 데이터 정합성
- **동일 요금제 방지**: 기존과 신규가 동일한 경우 변경 불가
- **최소 사용 기간**: 조기 변경/해지 방지
- **변경 횟수 제한**: 과도한 변경으로 인한 시스템 부하 방지
- **약정 정보**: 위약금 계산의 정확성 보장

### 7.8 에러 처리 전략
- **명확한 불가사유**: 각 체크 실패 시 구체적인 사유 제공
- **조기 반환**: 첫 번째 실패 조건에서 즉시 반환
- **경고 vs 불가**: 위약금은 경고, 조건 미충족은 불가
- **안내 메시지**: 요금 절감 등 긍정적 정보도 제공

### 7.9 고객 보호
- **요금 변동 고지**: 30% 이상 변동 시 명확한 안내
- **위약금 안내**: 약정 위약금 발생 시 금액 명시
- **부가서비스 안내**: 자동 해지되는 부가서비스 건수 안내
- **최소 사용 기간**: 조기 해지로 인한 불이익 방지
